# 🟩 Stage 1: Data Architecture & Typo-Immune Search

### Component 1.1: `load_and_clean_dataset`

* **Purpose:** Initializes the system by parsing the raw tracks feature file into an optimized in-memory data frame.
* **Specifications Checklist:**
* [ ] **Fault Tolerance:** Must use an engine configuration (`engine='python'`) that skips truncated rows (`on_bad_lines='skip'`) instead of crashing.
* [ ] **Prefix Indexing:** Must systematically pre-compute and store two-character lowercase shorthand index strings (`name_prefix` and `artist_prefix`) as explicit metadata columns.
* [ ] **Search Text Unification:** Must merge the track name and cleaned artist array into a standalone, lowercase `search_text` column (`'title - artist'`) to facilitate universal pattern matching.



### Component 1.2: `flexible_search`

* **Purpose:** Takes an arbitrary, potentially messy user text input and accurately evaluates matches using a highly optimized, dual-layer processing pipeline.
* **Specifications Checklist:**
* [ ] **String Sanitation:** Must strip trailing whitespace and enforce lowercase consistency on all raw queries.
* [ ] **100% Match Short-Circuit:** Must compare queries against both the raw track name and the combined `search_text` column. If a match is found, it must immediately exit and return a list containing just that record.
* [ ] **Prefix-Bucket Isolation:** If no exact match is found, it must extract the query's first two characters and isolate only that specific prefix partition from the database.
* [ ] **Dynamic Partition Expansion:** If a 2-character prefix bucket contains fewer than 10 candidate rows, it must automatically widen the bucket scope to a 1-character prefix fallback.
* [ ] **Fuzzy String Score Ranking:** Must use `thefuzz.process.extract` to score and rank the top 5 closest matches exclusively within the isolated partition bucket.
* [ ] **Output Uniformity:** Whether hitting the short-circuit or the fallback fuzzy track loop, it must return a uniform data structure: a nested Python list of pairs: `[['Song Name', 'Artist Name'], ...]`.

# 🟨 Stage 2: Core Vector Recommendation Engine

### Component 2.1: `calculate_taste_vector`

* **Purpose:** Processes a history list of explicit user song ratings and maps them to a single multi-dimensional coordinate vector representing the user's targeted music profile.
* **Specifications Checklist:**
* [ ] **Feature Slicing:** Must isolate and extract the 9 core acoustic dimensions (`danceability`, `energy`, `loudness`, `speechiness`, `acousticness`, `instrumentalness`, `liveness`, `valence`, `tempo`) for every song.
* [ ] **Range Normalization:** Must scale structural values (like `tempo` and `loudness`) so they match the 0.0 to 1.0 distribution range of the other attributes.
* [ ] **Explicit Rating Weighting Matrix:** Must convert 1-to-5 star user scores into proportional mathematical multiplier weights (e.g., highly weighted values for 5 stars, heavy negative penalties for 1 star).
* [ ] **Dot Product Matrix Summation:** Must compute the mathematical dot product of those rating weights against the isolated 9-feature row arrays to output a single, 9-element NumPy array taste vector.



### Component 2.2: `generate_recommendations`

* **Purpose:** Scans candidate tracks across the dataset using spatial geometry and similarity math to select optimal matching track outputs.
* **Specifications Checklist:**
* [ ] **Candidate Pool Sampling:** Must efficiently extract a diverse comparison database matrix of 50,000 highly indexed baseline rows.
* [ ] **Spatial Similarity Calculation:** Must calculate the spatial Cosine Similarity between the user's generated taste vector and every candidate record in the comparison matrix.
* [ ] **100% Acoustic Twin Safeguard:** Must evaluate the resulting similarity scores array. If any item scores $\ge 0.9999$ (a flawless acoustic match), it must trigger an absolute safety short-circuit.
* [ ] **Twin Short-Circuit Return:** If triggered, it must halt normal execution, discard all other results, and return *only* that single track object to the pipeline.
* [ ] **Sorted Output Construction:** If no identical twin is found, it must sort the candidate rows in descending order based on their similarity index and slice out the top 5 unique track identifiers.

# 🟦 Stage 3: Interactive Sandbox & CLI Simulator

### Component 3.1: `get_onboarding_page`

* **Purpose:** Manages data presentation slices to let users view and interact with the onboarding catalog without overwhelming the interface screen.
* **Specifications Checklist:**
* [ ] **Keyword Isolation:** Parameter signatures must never use reserved words (use `song_list` or `catalog`, never name a variable `list`).
* [ ] **Dynamic Index Bounds:** Must calculate bounds dynamically using `start_idx = page_number * 10` and `end_idx = min(start_idx + 10, len(song_list))` to guarantee it never overshoots the array edges.
* [ ] **Interface Context Syncing:** The print headers must accurately dynamically adapt to the available length of the sliced list (e.g., if a list finishes at 45 items, it must print `Songs 41 to 45` instead of `41 to 50`).



### Component 3.2: `run_local_simulator`

* **Purpose:** The central `while True:` execution loop that coordinates tracking memory states, navigation inputs, and system hand-offs directly in the terminal window.
* **Specifications Checklist:**
* [ ] **State Variable Integrity:** Must initialize clean session memory structures (`current_page = 0`, `user_ratings_db = []`) before entering the loop.
* [ ] **Navigation Interception:** Must accurately increment or decrement the `current_page` index variable when a user types strings like `"next"` or `"prev"`.
* [ ] **Floor/Ceiling Bounds Enforcement:** Must prevent navigation indices from crashing if a user tries to move past page boundaries (e.g., typing `"prev"` on Page 1 must keep `current_page == 0`).
* [ ] **Selection Entry Validation:** When a user types a specific song number from the menu, it must spawn a sub-loop that strictly validates that their rating is an integer between 1 and 5.
* [ ] **History Storage Structuring:** Must append validated inputs as clean track-score tuples directly into the `user_ratings_db` list.
* [ ] **Hand-Off Pipeline Transfer:** When `"done"` is caught, it must break the terminal loop, terminate navigation tracking, and pass the finished `user_ratings_db` array directly into Component 2.1.

# 🟨 Stage 4: Live Spotify API Integration Upgrade

### Component 4.1: `connect_spotify_client`

* **Purpose:** Secures, initializes, and maintains an active authenticated server-to-server connection with the live Spotify Web API ecosystem.
* **Specifications Checklist:**
* [ ] **Credential Decoupling:** Must load `client_id` and `client_secret` tokens through background system environment files, never via hardcoded strings inside code files.
* [ ] **Manager Initialization:** Must instantiate an automated `SpotifyClientCredentials` authorization manager object via the `spotipy` library wrapper.



### Component 4.2: `spotify_realtime_search`

* **Purpose:** Completely substitutes the local CSV prefix search pipeline with real-time web network calls to fetch trending tracks, artwork, and unique metadata identifiers.
* **Specifications Checklist:**
* [ ] **Network Request Swapping:** Replaces Component 1.2's DataFrame slice lookups with a live network query execution using `sp.search(q=query, type='track', limit=5)`.
* [ ] **Primary Key Extraction:** Must cleanly parse the incoming API JSON payload structure to isolate and save alphanumeric **Spotify Track IDs** to use as universal primary keys.
* [ ] **Upstream Regional Market Injection:** Must accept a string parameter token (e.g., `market='KR'` or `market='DE'`) to restrict live search results to specific global markets.



### Component 4.3: `spotify_feature_fetch`

* **Purpose:** Pulls precise live acoustic coordinate features for any song on earth directly from Spotify's servers using unique track keys.
* **Specifications Checklist:**
* [ ] **Endpoint Target Routing:** Must map Track IDs directly to Spotify’s `sp.audio_features(track_ids)` endpoint.
* [ ] **Float Array Data Parsing:** Must convert the returning API dictionary data into the standardized 9-dimensional float structure required by Component 2.1.
* [ ] **Downstream Metadata Filtering:** Must query artist profile endpoints and scan the `genres` array to enforce strict metadata filter matches (e.g., confirming `k-pop` is present) before letting recommendations pass to the interface.

# 🚀 Stage 5: Live Telegram Bot Deployment

### Component 5.1: `telegram_polling_engine`

* **Purpose:** Establishes an asynchronous, event-driven network handler that listens for incoming chat traffic and securely pathways it to appropriate internal functions.
* **Specifications Checklist:**
* [ ] **Asynchronous Framework Boot:** Uses `python-telegram-bot` to spin up a background application listener thread without blocking main thread operations.
* [ ] **Isolated State Tree Storage:** Must instantiate a central dict system indexed strictly by individual `user_id` values to prevent multi-user database cross-talk or runtime variable leaks.
* [ ] **Routing Assignment:** Must deploy explicit message handlers (`CommandHandler`, `CallbackQueryHandler`) to separate text string commands from physical UI button clicks.



### Component 5.2: `render_inline_interface`

* **Purpose:** Translates raw Python lists into interactive, clickable button rows on user chat interfaces.
* **Specifications Checklist:**
* [ ] **Markup Layout Matrix Mapping:** Must loop through data arrays and convert plain strings into physical `InlineKeyboardButton` objects.
* [ ] **Payload String Serialization:** Every UI button must embed a discrete background payload data tag (`callback_data`) containing an explicit command and track ID index token (e.g., `"select_song_2"`).



### Component 5.3: `handle_interface_callback`

* **Purpose:** Catches background tap signals from phone screens, adjusts user state metrics, and refreshes interfaces in real time.
* **Specifications Checklist:**
* [ ] **Payload Signal De-serialization:** Must capture incoming `callback_data` strings and split them to isolate the true operation directive and parameters.
* [ ] **Non-Blocking In-Place Editing:** Must overwrite existing chat bubbles instantly using `edit_message_reply_markup` to cycle onboarding pages seamlessly without spamming new messages.
* [ ] **Final Recommendation Display Execution:** When catching a completed state signal, it must pass session tracking histories to Stage 2 and return a beautiful final recommendation block complete with album art and clickable streaming links.
